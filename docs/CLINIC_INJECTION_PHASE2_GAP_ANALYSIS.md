# Clinic Injection Tokens & Injection Room — Phase 2 Gap Analysis

**Purpose:** Operational completeness, auditability, and scalability after Phase 1 enterprise upgrade.  
**Scope:** Token generation hardening, audit/compliance, injection room operations, scalability, implementation plan.  
**Analysis only — no code changes.**

---

## 1. TOKEN GENERATION HARDENING

| Capability | Status | Notes |
|------------|--------|-------|
| **Visit search / typeahead** | **Missing** | Backend `listVisits` supports only `petId`, `patientId`, `limit`, `offset`. No search by treatmentCode, visit ID range, date range, or free text. Frontend uses “Visit ID + Verify” only. |
| **Prescription line selection** | **Missing** | Backend generate accepts `prescriptionId`; service validates prescription belongs to visit. No API to list prescriptions by visit with line items for token generation. Frontend does not send prescriptionId. |
| **Treatment course selection** | **Partial** | Backend accepts `treatmentCourseId`/`treatmentDayId`; `listTreatmentCourses(branchId, { patientId, status, skip, take })` exists. Frontend generate form does not expose course/day selectors (no patient-in-context from visit used to load courses). |
| **Treatment day selection** | **Partial** | Same as above; days are under a course. No dedicated “list days for course” or “next due day” API in token flow. |
| **Selected vial session selection** | **Partial** | Backend accepts `selectedVialSessionId`; `listVialSessions(branchId, { variantId, status })` exists. Frontend generate form does not show vial dropdown (injection-room does, post-validate). |
| **Source rules and validation** | **Implemented** | Backend: paid order required, visit in branch, variant exists, prescription optional validation. Branch policy `tokenValiditySameDayOnly` for expiry. No frontend-specific source rules beyond required fields. |

**Summary:** Token generation is **visit + variant + dose** with optional backend fields unused in the UI. Workflow-aware form would need: visit search/typeahead, prescriptions-by-visit (with line selection), treatment course/day selectors, vial session selector for variant, and optional order selector — plus keeping all validation on the backend.

## 2. AUDIT AND COMPLIANCE GAP

### 2.1 What is recorded today

| Event | DB / backend | Stored today |
|-------|--------------|--------------|
| Token generated by | InjectionToken.`generatedByUserId` | Yes |
| Token used by | InjectionToken.`usedByUserId`, `usedAt` | Yes |
| Token cancelled by | Reuses `usedByUserId`/`usedAt` for cancel (semantic overload) | Partial — no distinct “cancelled by” or “cancelled at” |
| Token validated by | Not persisted | **Missing** — validate is read-only, no audit row |
| Cancel reason | Not in schema | **Missing** — cancel has no reason field |
| Token status transition history | No status log table | **Missing** — only current status on InjectionToken |
| Emergency bypass reason | Not in schema; frontend collects but does not send | **Missing** — recordDose body has no `emergencyBypassReason` |
| Override / approval linkage | MedicineApprovalRequest has `relatedEntityType`/`relatedEntityId`; no FK from MedicationAdministration | **Missing** — dose record not linked to approval when override used |
| Dose record audit trail | MedicationAdministration has `administeredByUserId`, `administeredAt`; no clinic audit log for dose | Partial — entity row exists; no writeClinicAudit for dose/token |

### 2.2 Missing DB fields (Prisma)

- **InjectionToken:** `cancelledByUserId Int?`, `cancelledAt DateTime?`, `cancelReason String? @db.Text`. (Optional: stop reusing usedByUserId for cancel.)
- **InjectionToken:** No “validated” tracking — if needed for compliance, add `validatedByUserId Int?`, `validatedAt DateTime?` (and optionally persist on first validate).
- **MedicationAdministration:** `emergencyBypassReason String? @db.Text`, `medicineApprovalRequestId Int?` (optional FK to MedicineApprovalRequest when dose is under override).

### 2.3 Missing backend handlers / services

- **Cancel:** Accept optional `reason` in PATCH body; pass to service; persist `cancelledByUserId`, `cancelledAt`, `cancelReason` (after migration).
- **Validate:** Optionally record validation event (new audit table or clinic audit log) with userId, tokenId, timestamp — requires clinic.audit extension for INJECTION_TOKEN entity/actions.
- **Record dose:** Accept optional `emergencyBypassReason` in POST body; persist on MedicationAdministration (after migration). Optionally accept `medicineApprovalRequestId` when recording under an approved override.

### 2.4 Clinic audit extension

- **clinic.audit:** Add actions e.g. `INJECTION_TOKEN_GENERATED`, `INJECTION_TOKEN_VALIDATED`, `INJECTION_TOKEN_CANCELLED`, `INJECTION_TOKEN_USED`, `MEDICATION_ADMINISTERED` and entity types `INJECTION_TOKEN`, `MEDICATION_ADMINISTRATION`. Call `writeClinicAudit` from controller (or service) for generate, cancel, validate (if logged), recordDose.
- **Audit writer:** Ensure entity type is accepted (AuditEntityType or generic) for these new types.

### 2.5 Token status transition history

- **Option A:** New table `InjectionTokenStatusLog` (tokenId, fromStatus, toStatus, changedByUserId, changedAt, reason?) and write a row on every status change (PENDING→USED, PENDING→CANCELLED, PENDING→EXPIRED).
- **Option B:** Rely on clinic audit log entries (action + entityType + entityId + after) so status history is reconstructable from audit. Prefer B if audit log is retained and queryable by entity.

## 3. INJECTION ROOM OPERATIONS GAP

### 3.1 Current backend/frontend support

| Capability | Backend | Frontend | Notes |
|------------|---------|----------|-------|
| **Queue of pending tokens** | **Partial** | **Missing** | `getInjectionMonitoringDashboard` returns `pendingTokens` (PENDING tokens, take 50). No dedicated “injection room queue” API. Frontend injection-room does not show a queue; user enters token code. |
| **Validated tokens “ready for injection”** | **Missing** | N/A | No “validated at” or “ready” state; validate is stateless. Could treat “recently validated in this session” as UI-only “ready” list. |
| **Completed today** | **Partial** | **Missing** | Dashboard has injectionsToday count and administrations by source; no dedicated “list doses completed today” for injection-room. Dose-by-visit exists; “doses today by branch” would require new endpoint or reuse dashboard aggregation. |
| **Bypass cases** | **Partial** | **Implemented** | Backend allows recordDose with emergencyBypass; no stored reason. Frontend modal collects reason (not sent). No “list bypass doses” API. |
| **Expired/rejected tokens** | **Supported** | **Partial** | List tokens supports status=EXPIRED; validate returns error for expired/used. No dedicated “rejected today” or “expired today” list for room. |
| **Room-based filtering or enforcement** | **Missing** | **Missing** | InjectionToken and MedicationAdministration have no roomId. VialSession has roomId. Room could be inferred from vial session at dose time; no filter by room on token or dose list. |

### 3.2 Proposed architecture for injection room operations

- **Queue API:** Add GET `medicine-control/injection-room/queue` (or reuse injection-monitor dashboard with a slice): branchId, optional `roomId`, optional `date`; return `{ pendingTokens: Token[], completedToday: number, bypassCountToday: number }`. Pending tokens = listTokens(status=PENDING, branchId) with optional room filter if tokens ever get a room hint (e.g. from selectedVialSession.roomId).
- **Room enforcement:** Option A — add `roomId` to MedicationAdministration (and optionally to InjectionToken when generated in a room context). Option B — keep dose room-agnostic and derive “injections in room X” from vial sessions (roomId on VialSession) joined to administrations. Option B avoids schema change and matches “room is where the vial is” model.
- **Completed today / bypass list:** New endpoint GET `medicine-control/dose/list` with query `branchId`, `date`, `emergencyBypassOnly?: boolean` returning list of administrations for that branch/date (and optionally only bypass). Alternatively extend injection-monitor dashboard response to include `administrationsToday[]` (or a summary) so injection-room can show “completed today” without a new endpoint.
- **Validated-ready:** No DB change needed. Frontend can keep a “recently validated” list in component state (e.g. last N validated token codes) and show a small “Ready to inject” list that clears on refresh or “Start over.”

## 4. SCALABILITY GAP

| Area | Current | Gap |
|------|---------|-----|
| **List pagination** | Backend: skip/take, take capped at 100. Frontend: take=100, no skip; no “Load more” or page controls. | **Partial** — backend supports pagination; frontend does not use skip or show pages. |
| **Filter completeness** | Tokens: status, tokenCode, fromDate, toDate, visitId, patientId. Visits: petId, patientId, limit, offset (no date, no treatmentCode search). | Tokens filters are adequate; visits lack search/date filters for token generation. |
| **Search performance** | tokenCode uses `contains` + mode insensitive; no index on tokenCode for prefix. List visits ordered by createdAt desc; no full-text search. | Consider index on tokenCode if prefix search is needed; visit search may need treatmentCode/date indexes. |
| **Table limits** | Backend listTokens take max 100; listVialSessions take max 100; listVisits limit max 100. | Adequate for single-branch; document limits. |
| **Response shape consistency** | List endpoints return `{ list, total }`; dashboard returns nested objects. Frontend normalizes. | **Implemented** — consistent. |
| **Typed API contract consistency** | Frontend has `clinicMedicineControl.ts` types; API methods return `any` or generic Promise. Backend has no shared OpenAPI/contract. | **Partial** — frontend types exist; backend DTOs not exported; no single contract file. |

**Recommendations:** Add pagination UI (skip/take or “Load more”) on injection-tokens list; keep take cap 100. Add visit search (e.g. treatmentCode, date range) in backend listVisits and use in token generation. Optionally add token list stats endpoint (counts by status) to avoid loading full list for stats. Document API limits in one place.

## 5. IMPLEMENTATION PLAN (Phase 2)

### A. Backend changes (order)

1. **DB/Prisma (migration)**  
   - InjectionToken: add `cancelledByUserId Int?`, `cancelledAt DateTime?`, `cancelReason String? @db.Text`. (Decide whether to keep using usedByUserId for cancel or use new fields only.)  
   - MedicationAdministration: add `emergencyBypassReason String? @db.Text`, optional `medicineApprovalRequestId Int?` (FK to MedicineApprovalRequest).  
   - Optional: InjectionTokenStatusLog or rely on audit only.

2. **Clinic audit**  
   - Extend clinic.audit with actions and entity types for INJECTION_TOKEN and MEDICATION_ADMINISTRATION.  
   - Call writeClinicAudit in generateInjectionToken, cancelInjectionToken, recordDose; optionally in validateInjectionToken (if “validated by” is logged).

3. **Cancel token**  
   - PATCH body: optional `reason`.  
   - injectionToken.service: cancelToken accepts reason; write cancelledByUserId, cancelledAt, cancelReason (new fields).

4. **Record dose**  
   - POST body: optional `emergencyBypassReason`, optional `medicineApprovalRequestId`.  
   - doseConsumption.service: persist on MedicationAdministration.

5. **Validate (optional)**  
   - If “validated by” is required: add validatedByUserId/validatedAt on InjectionToken and set on first successful validate (or log only via audit).

6. **Injection room queue / completed today**  
   - New GET `medicine-control/injection-room/queue` or extend injection-monitor: return pendingTokens (existing), plus completedToday count and optionally list of administrations today (or bypass-only).  
   - Or: GET `medicine-control/dose/list?branchId=&date=&emergencyBypassOnly=` for dose list by branch/date.

7. **Visit search**  
   - listVisits: add optional query params `treatmentCode`, `fromDate`, `toDate`, `search` (e.g. treatmentCode or patient name substring). emr.service listVisits extended.

### B. Frontend changes (order)

1. **Types**  
   - Update clinicMedicineControl types for new fields (cancelReason, emergencyBypassReason, etc.).  
   - Ensure API layer uses typed request/response where possible.

2. **Token generation hardening**  
   - Add visit search/typeahead using listVisits (with new params if added): search by treatmentCode or date range, select visit from list.  
   - Optional: after visit selected, load prescriptions and treatment courses for that visit/patient; add prescription dropdown and treatment course/day dropdown; add vial session dropdown for selected variant.  
   - Keep validation rules on backend; frontend only sends selected ids.

3. **Injection tokens page**  
   - Cancel modal: add “Reason for cancel” (optional or required by policy); send in PATCH body.  
   - Pagination: add skip/take or “Load more” for token list; show page size selector if needed.

4. **Injection room page**  
   - Emergency bypass: send `emergencyBypassReason` in recordDose body (after backend supports it).  
   - Optional: “Queue” tab or panel that calls new queue API and shows pending tokens; “Completed today” from new endpoint or dashboard slice.

5. **Token detail drawer**  
   - Show generatedBy (and if added: validatedBy, cancelledBy, cancelReason) from API; ensure list/detail include these after backend returns them.

### C. Database / Prisma changes (summary)

- Migration 1: InjectionToken — add cancelledByUserId, cancelledAt, cancelReason.  
- Migration 2: MedicationAdministration — add emergencyBypassReason, medicineApprovalRequestId (optional FK).  
- No new tables required if status history is covered by clinic audit log.

### D. QA scenarios

- Generate token with visit search (by treatmentCode/date).  
- Generate with prescription and treatment course/day when UI added.  
- Cancel token with reason; verify cancelledByUserId, cancelledAt, cancelReason stored and shown.  
- Record dose with emergency bypass and reason; verify emergencyBypassReason stored.  
- Record dose linked to override (medicineApprovalRequestId) when applicable.  
- Audit log: generate, cancel, record dose produce expected clinic audit entries.  
- Injection room queue: pending list and completed today (and bypass count) when API added.  
- Pagination: token list with skip/take; large result set.  
- Room filter: if implemented, filter queue/doses by room.

## 6. Architecture note

**Extending current pages vs new structure:** The current architecture (injection-tokens and injection-room as two pages under medicine-control, with shared types and dashboard components) is sound. Phase 2 should **extend these pages and the same backend services** rather than introduce a parallel flow:

- Add fields and APIs as above; keep business rules in backend (generateToken, validateToken, cancelToken, recordAdministration).  
- Frontend stays thin (forms, validation UX, display); no duplication of rules.  
- Reusable components (PageHeader, StatCard, DetailDrawer, StatusBadge, modals) stay; add only what’s needed (e.g. VisitSearchSelect, TokenQueuePanel) in shared or page-local components.  
- Typed API contracts: extend existing `clinicMedicineControl` types and backend service types; optional OpenAPI later.

A separate “injection room app” or duplicate routes would add maintenance and inconsistency; avoid.

### 4. Risk list

| Risk | Mitigation |
|------|------------|
| Breaking existing cancel flow | Keep backward compatibility: cancel without reason still works; usedByUserId/usedAt for cancel can remain for backward read; new fields additive. |
| Audit log volume | Clinic audit may grow; ensure retention and indexing by entityType/entityId. |
| Visit search performance | Add indexes on visit.treatmentCode, visit.createdAt, and optional patient display name if search by name. |
| Frontend sending reason to unupdated backend | Deploy backend first; frontend can send reason only when supported (backend ignores unknown fields until implemented). |

6. **Frontend: injection-tokens** — cancel modal reason; pagination.  
7. **Frontend: injection-room** — send emergencyBypassReason.  
8. **Backend: visit search** — extend listVisits; then frontend visit search/typeahead.  
9. **Backend (optional): queue/dose list API** — implement and expose.  
10. **Frontend (optional): queue panel, completed today** — consume new API.  
11. **QA** — run through checklist (cancel with reason, bypass with reason, audit entries, pagination, visit search, queue).
