## Summary of completed work

### A. Operator accountability

- **List tokens:** Query params validatedByMe=true, generatedByMe=true (resolved server-side to req.user.id). List response includes validatedBy, generatedBy (displayName). Injection-tokens page: "All operators" / "Validated by me" / "Generated by me" dropdown.
- **Board:** Optional validatedByMe filter for pending and unassigned; pending tokens show validatedBy badge; completed today includes and displays administeredBy.

### B. Room execution completion

- **Unassigned bucket:** Board returns unassignedTokens (pending tokens with no selectedVialSessionId). When roomId is set, pendingTokens = in-room only, unassignedTokens = no-vial. Injection-room page: "Unassigned (no room)" tab.
- **Room mismatch:** Backend recordDose rejects with 400 ROOM_MISMATCH when token had a pre-selected vial in room A and the submitted vialSessionId is in room B. Frontend: Step 3 shows warning when selected vial room ≠ token vial room; vial dropdown and pre-selected vial show room name; submit error shows friendly message.

### C. Consistency hardening

- **Token context:** selectedVialSession now includes room (id, name, code) for room mismatch UX.
- **Dose consistency:** Step 4 shows "Expected from token: X ml" and an info alert when administered dose differs from token expected dose (confirm intent).
- **Submit payload:** Unchanged; backend enforces variant/visit and room when applicable.

---

## Remaining known gaps

- **"My completed injections today" filter:** Board does not filter completedToday by administeredByUserId; can be added in a later iteration if needed.
- **Explicit room on token:** Room remains derived from selectedVialSession only; no roomId column on InjectionToken.
- **Operator in token list table:** List includes validatedBy/generatedBy; table columns could show "Validated by" / "Generated by" (currently in drawer/detail).
