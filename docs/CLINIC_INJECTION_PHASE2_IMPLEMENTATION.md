## Summary of completed work

### A. Token generation hardening
- **Visit search / typeahead:** Staff can search visits by treatment code or patient/pet name; results in a dropdown; selecting a visit sets visit ID and summary. Backend listVisits supports `treatmentCode`, `fromDate`, `toDate`, `search`.
- **Cancel reason:** Cancel modal includes an optional “Reason for cancellation” textarea; value sent to PATCH and stored in `cancelReason`.
- **Prescription / treatment course / day / vial:** Backend already accepted prescriptionId, treatmentCourseId, treatmentDayId, selectedVialSessionId; frontend generate form still uses visit + variant + dose (no prescription/course/day/vial selectors in this phase to keep the form dense).

### B. Audit and compliance
- **Cancel:** `cancelledByUserId`, `cancelledAt`, `cancelReason` persisted; token detail drawer shows “Cancelled by” and “Reason” (no raw JSON).
- **Emergency bypass reason:** Request body and MedicationAdministration.emergencyBypassReason; injection-room bypass modal and token-flow bypass send reason.
- **Clinic audit:** writeClinicAudit for INJECTION_TOKEN_GENERATED, INJECTION_TOKEN_CANCELLED, MEDICATION_ADMINISTERED; entity types INJECTION_TOKEN, MEDICATION_ADMINISTRATION.
- **Token detail drawer:** Audit block shows Generated by, Used by, Cancelled by (with time), Cancel reason.

### C. Injection room operations board
- **Board API:** GET `medicine-control/injection-room/board` returns pendingTokens, completedToday, bypassToday, expiredOrProblemToday for the branch (optional date).
- **Frontend:** Operations board card with tabs: Pending/ready, Completed today, Bypass cases, Expired/problem; lists from board API; board refreshes after recording a dose.

### D. Scalability
- **Token list pagination:** skip/take (page size 20), Previous/Next, “Page X of Y”; filters reset page to 0.
- **Visit search:** listVisits supports treatmentCode, fromDate, toDate, search; typeahead uses search with limit 20.
- **Typed contracts:** Frontend types and API payloads updated for new fields and InjectionRoomBoard.

---

## Remaining known gaps

- **Token generation:** Prescription line selector, treatment course/day selectors, and vial session selector are not in the generate form (backend supports them); add in a follow-up if needed.
- **Token status transition history:** No dedicated InjectionTokenStatusLog table; history is inferable from clinic audit log (action + entityType + entityId + after).
- **Validated-by:** Validate is still read-only; no validatedByUserId/validatedAt on token or in audit (optional future addition).
- **Room-based filtering:** Board and tokens are branch-scoped; no roomId on token/dose; room can be derived from vial session when needed.
- **Dose list by date:** Board provides “completed today” and “bypass today”; no separate GET dose/list by branch+date beyond the board payload.
